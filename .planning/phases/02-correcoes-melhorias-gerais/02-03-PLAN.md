# Plan 02-03: Providers LLM — Cerebras, Mistral, Groq, OpenRouter

## Goal

Quatro ajustes pontuais nos providers do `llm.service.ts` (backend) e no catálogo de modelos do `FloatingChat.tsx` (frontend):

1. **Groq:** Remover `qwen-2.5-32b` (descontinuado) e corrigir `contextWindow` do DeepSeek R1 (128000 está incorreto).
2. **Cerebras:** Aumentar `max_tokens` padrão de 2048 para 8192 quando o provider for Cerebras (ambos os modelos suportam 8192).
3. **OpenRouter:** Adicionar tratamento de fallback de modelo (404 → próxima opção na lista free) e sincronizar o catálogo frontend com os modelos disponíveis.
4. **Frontend (FloatingChat):** Sincronizar o array `MODELS` com o catálogo backend atualizado e adicionar modelos DeepSeek/Gemma que estavam ausentes.

## Context

**Groq (BUG-05 parcial):** `GROQ_MODELS` inclui `qwen-2.5-32b` que pode não estar mais disponível na API Groq. O `deepseek-r1-distill-llama-70b` tem `contextWindow: 128000` declarado mas o limite real na Groq é menor (≈32768 tokens de contexto para esse modelo). Manter valor errado causa prompts truncados silenciosamente.

**Cerebras:** `body.max_tokens` default é 2048 (linha 584 do `llm.service.ts`). Cerebras suporta até 8192. Respostas longas (geração de portaria) ficam truncadas. Solução: quando `provider === 'cerebras'` e `options.maxTokens` não foi especificado, usar 8192.

**OpenRouter:** A lista `OPENROUTER_FREE_MODELS` tem 3 modelos free. Se o primeiro (`meta-llama/llama-3.3-70b-instruct:free`) retornar 404, o circuit breaker atual marca a chave como esgotada (cooldown 5min via `markKeyExhausted`) — mas a chave está válida, só o modelo mudou. O tratamento correto para 404 de modelo inexistente é tentar o próximo modelo free, não marcar a chave como esgotada.

**Frontend:** O array `MODELS` em `FloatingChat.tsx` não inclui `google/gemma-3-27b-it:free` e `deepseek/deepseek-r1:free` como opções selecionáveis pelo usuário (apenas usados como fallback interno). Adicionar ao seletor com label adequado.

## Tasks

### Task 1: Atualizar GROQ_MODELS e ajustar max_tokens Cerebras

**File:** `apps/api/src/services/llm.service.ts`

**What:**
- Remover `qwen-2.5-32b` do array `GROQ_MODELS`
- Corrigir `contextWindow` do `deepseek-r1-distill-llama-70b` de `128000` para `32768`
- Ajustar `max_tokens` para Cerebras: quando `provider === 'cerebras'` e `options.maxTokens` não foi definido, usar `8192` em vez do padrão `2048`

**Why:** Modelos descontinuados retornam 404 causando falha desnecessária do provider inteiro. `contextWindow` incorreto leva o smart router a enviar prompts maiores do que o modelo suporta, resultando em truncamento ou erro. Cerebras suporta 8192 tokens de saída — usar o máximo evita corte em respostas longas.

**How:**
1. Localizar `GROQ_MODELS` (linha ~79) e remover a entrada `qwen-2.5-32b`:
   ```typescript
   // REMOVER esta linha:
   { id: 'qwen-2.5-32b', label: 'Qwen 2.5 32B (Raciocínio)', contextWindow: 32768 },
   ```
2. Corrigir `contextWindow` do DeepSeek R1 para `32768`:
   ```typescript
   { id: 'deepseek-r1-distill-llama-70b', label: 'DeepSeek R1 (Contexto longo)', contextWindow: 32768 },
   ```
3. Localizar a construção do `body` em `callProviderRaw` (linha ~581-586):
   ```typescript
   const body: any = {
       model,
       messages: normalizedMessages,
       max_tokens: options.maxTokens ?? 2048,
       ...
   }
   ```
   Modificar para:
   ```typescript
   const defaultMaxTokens = provider === 'cerebras' ? 8192 : 2048
   const body: any = {
       model,
       messages: normalizedMessages,
       max_tokens: options.maxTokens ?? defaultMaxTokens,
       ...
   }
   ```

### Task 2: Melhorar tratamento de 404 de modelo no OpenRouter

**File:** `apps/api/src/services/llm.service.ts`

**What:** No bloco de tratamento de erros dentro de `callWithCircuitBreaker`, quando o provider é `openrouter` e o status é 404, não marcar a chave como esgotada — apenas retornar `skipReason: 'api_error'` para que o fallback do caller tente o próximo modelo free.

**Why:** O 404 do OpenRouter para modelo inexistente/removido da tier free não indica que a chave API é inválida. Marcar como `esgotada` via `markKeyExhausted` coloca a chave em cooldown de 5 minutos desnecessariamente, impedindo que o provider seja usado para outros modelos.

**How:**
1. Localizar o bloco de catch dentro de `callWithCircuitBreaker` (aprox. linha 364-386)
2. Encontrar o trecho:
   ```typescript
   if (status === 400 || status === 404) {
       return { tried: false, skipReason: 'api_error' } as any
   }
   ```
3. Modificar para diferenciar OpenRouter:
   ```typescript
   if (status === 404 && provider === 'openrouter') {
       // Modelo não disponível (saiu da tier free) — chave válida, pular sem cooldown
       console.warn(`[LLM Pool] OpenRouter modelo '${model}' não disponível (404) → sem cooldown na chave`)
       return { tried: false, skipReason: 'api_error' } as any
   }
   if (status === 400 || status === 404) {
       return { tried: false, skipReason: 'api_error' } as any
   }
   ```
   Também adicionar: antes do `markKeyExhausted(key.id, 5)` na linha ~380, adicionar guard para não fazer cooldown em 404 de OpenRouter:
   ```typescript
   if (!(status === 404 && provider === 'openrouter')) {
       await markKeyExhausted(key.id, 5)
   }
   ```
