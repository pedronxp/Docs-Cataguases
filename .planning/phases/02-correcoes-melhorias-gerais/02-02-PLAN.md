# Plan 02-02: Notificações + Fila de Aprovação

## Goal

Dois bugs de fluxo independentes:

1. Uma única ação `ASSUMIR_REVISAO` gera 3 registros no banco (2 `FeedAtividade` + 1 `Notificacao`) que chegam ao frontend via SSE em sequência, produzindo 3 toasts e 3 badges de notificação para 1 ação humana.
2. A fila de aprovação de usuários (`/admin/fila-aprovacao`) tem comportamento quebrado — o backend PATCH `/api/admin/users/:id` foi inspecionado e parece correto, mas a UI pode estar enviando payload sem campo obrigatório `secretariaId` para roles que exigem.

## Context

**Notificações 3x (BUG-03):** Em `fluxo/route.ts` linhas 223-235, para `action === 'ASSUMIR_REVISAO'` ou `action === 'SOLICITAR_REVISAO'`, um segundo `FeedAtividade` com `tipoEvento: 'REVISAO_ATRIBUIDA'` é criado além do principal já gerado nas linhas 196-205 (`tipoEvento: 'MUDANCA_STATUS_SOLICITAR_REVISAO'`). Após a transação, ainda é criada uma `Notificacao` via `criarNotificacao` (linha 260). O SSE emite todos os 3 como eventos separados. A deduplicação no store usa somente `id` — como `FeedAtividade.id` e `Notificacao.id` são registros distintos com IDs diferentes, a deduplicação não os filtra. Solução mínima: remover o bloco duplicado de `FeedAtividade` (linhas 223-235 no contexto `SOLICITAR_REVISAO || ASSUMIR_REVISAO`).

**Fila de Aprovação (BUG-04):** A UI em `_sistema.admin.fila-aprovacao.tsx` chama `atualizarUsuario(id, { role, secretariaId, setorId, ativo: true })`. O backend valida com `userUpdateSchema` que exige `secretariaId` quando `role` é `REVISOR | OPERADOR | SECRETARIO`. O fluxo parece correto no código inspecionado. Deve-se verificar o `UsuarioService.atualizarDadosAdmin` (backend) para confirmar que não tem guard extra causando falha silenciosa.

**Deduplicação aprimorada no store:** A deduplicação atual (`x.id === n.id`) só funciona para o mesmo registro. Adicionar deduplicação por `(portariaId + tipoEvento + timestamp ±10s)` elimina toasts duplicados mesmo quando chegam por fontes diferentes (SSE feed vs SSE notificacao).

## Tasks

### Task 1: Remover FeedAtividade duplicado no fluxo/route.ts

**File:** `apps/api/src/app/api/portarias/[id]/fluxo/route.ts`

**What:** Remover o bloco `if (action === 'SOLICITAR_REVISAO' || action === 'ASSUMIR_REVISAO')` que cria um segundo `FeedAtividade` com `tipoEvento: 'REVISAO_ATRIBUIDA'` (linhas ~223-235).

**Why:** O FeedAtividade principal com `tipoEvento: 'MUDANCA_STATUS_SOLICITAR_REVISAO'` (linhas 196-205) já registra a transição. O segundo é redundante e gera toast extra. A `Notificacao` direcionada ao autor (linhas ~258-265 após a transação) é mantida pois é persistida por usuário e não gera toast adicional via SSE se o store já a tiver.

**How:**
1. Abrir `apps/api/src/app/api/portarias/[id]/fluxo/route.ts`
2. Localizar o bloco dentro da transação `prisma.$transaction`:
   ```typescript
   // Notificação geral quando assumido
   if (action === 'SOLICITAR_REVISAO' || action === 'ASSUMIR_REVISAO') {
       await tx.feedAtividade.create({
           data: {
               tipoEvento: 'REVISAO_ATRIBUIDA',
               ...
           }
       })
   }
   ```
3. Remover esse bloco inteiro (o `if` e o `await tx.feedAtividade.create` dentro dele)
4. Manter o bloco de `REJEITAR_REVISAO || REJEITAR_CHEFIA` que cria `FeedAtividade` para `DOCUMENTO_DEVOLVIDO_AUTOR` — esse é correto (notifica o autor sobre devolução, não é duplicata)
5. Manter o bloco após a transação que chama `criarNotificacao` (linhas ~258-265) — é a `Notificacao` persistente direcionada ao autor

### Task 2: Aprimorar deduplicação no notifications.store.ts

**File:** `apps/web/src/store/notifications.store.ts`

**What:** Adicionar deduplicação semântica no método `addNotificacao` para filtrar notificações com mesmo `portariaId + tipoEvento` dentro de uma janela de 10 segundos, além da deduplicação por `id` já existente.

**Why:** `FeedAtividade` e `Notificacao` são registros diferentes com IDs diferentes mas representando o mesmo evento humano. A deduplicação semântica elimina o segundo toast mesmo que os IDs difiram.

**How:**
1. No método `addNotificacao` em `notifications.store.ts`, após a verificação de `id` duplicado (linha ~40), adicionar:
   ```typescript
   // Deduplicação semântica: mesmo portariaId + tipoEvento dentro de 10s
   if (n.portariaId && n.tipoEvento) {
       const JANELA_MS = 10_000
       const agora = Date.now()
       const jaExiste = state.notificacoes.some((x) => {
           if (x.portariaId !== n.portariaId) return false
           if (x.tipoEvento !== n.tipoEvento) return false
           const xTs = x.criadoEm ? new Date(x.criadoEm).getTime() : 0
           const nTs = n.criadoEm ? new Date(n.criadoEm).getTime() : agora
           return Math.abs(xTs - nTs) < JANELA_MS
       })
       if (jaExiste) return state
   }
   ```
2. A verificação de `id` duplicado existente deve ser mantida como primeira guarda (mais barata)

### Task 3: Corrigir invalidação de cache na fila de aprovação

**File:** `apps/web/src/routes/_sistema.admin.fila-aprovacao.tsx`

**What:** Após aprovar ou rejeitar um usuário, a lista de pendentes não atualiza corretamente porque o `refetch()` pode não estar invalidando o cache do TanStack Query. Corrigir a estratégia de invalidação para garantir que a lista some imediatamente após a ação.

**Why:** Backend e service foram inspecionados — ambos corretos. `UsuarioService.atualizarDadosAdmin` faz um `prisma.user.update` simples e retorna `{ ok, value }`. O route handler mapeia corretamente. O bug é de estado no frontend: `refetch()` re-executa a query mas pode estar usando cache stale. Remover o usuário aprovado/rejeitado otimisticamente da lista garante UX imediata independente do cache.

**How:**
1. Abrir `apps/web/src/routes/_sistema.admin.fila-aprovacao.tsx`
2. Localizar `confirmarAprovacao` (linha ~129) e `confirmarRejeicao` (linha ~149)
3. Em `confirmarAprovacao`, após `toast(...)` e antes de `setModalAprovar(null)`, adicionar remoção otimista da lista:
   ```typescript
   // Remove o usuário aprovado da lista imediatamente (otimista)
   setUsuarios(prev => prev.filter(u => u.id !== modalAprovar.id))
   ```
4. Em `confirmarRejeicao`, após `toast(...)` e antes de `setModalRejeitar(null)`, adicionar:
   ```typescript
   // Remove o usuário rejeitado da lista imediatamente (otimista)
   setUsuarios(prev => prev.filter(u => u.id !== modalRejeitar.id))
   ```
5. Verificar se `usuarios` vem de um `useState` local ou de um hook externo (`useUsuarios`). Se vier de hook externo com `useQuery`, usar `queryClient.invalidateQueries(['usuarios'])` em vez de `setUsuarios`. Ler o topo do arquivo para identificar a origem do estado.
6. Manter o `refetch()` existente como sincronização com o servidor (já presente no código)
