Plano de Implementação — Doc's Cataguases Auditoria completa + Roadmap de melhorias · Março 2026

1. Diagnóstico Atual (Auditoria)

Problemas identificados por área:

Notificações globais, não por usuário — O SSE filtra por secretaria/setor mas NÃO filtra por destinatário específico. Todos da mesma secretaria veem tudo. Não há notificação "para você" (ex: "sua portaria foi rejeitada").
Sidebar sem badges dinâmicos — A navbar lateral tem suporte a badge?: number mas nenhum item usa. Revisão não mostra contagem de itens pendentes.
Assinatura trust-based (mock) — O sistema marca portarias como "assinadas" sem verificação criptográfica real. SHA-256 usado apenas para integridade do PDF, não prova de assinatura. Sem ICP-Brasil.
Tela "Aguardando Aprovação" incompleta — A rota /aguardando existe mas falta: polling para verificar se foi aprovado, feedback visual do status, e redirecionamento automático após aprovação.
Sem auto-refresh (F5 obrigatório) — Dashboard, lista de portarias e fila de revisão fazem apenas 1 load no mount. Nenhum polling, nenhum refetch automático, nenhuma invalidação de cache.
Fluxo completo funcional mas com gaps — State machine bem estruturada, mas: sem rollback de PRONTO_PUBLICACAO, transferência limitada a REVISOR, sem timeout/SLA.
Chatbot IA: Failover silencioso troca modelo sem aviso — Quando Cerebras dá rate limit, o sistema faz fallback automático para Groq trocando o modelo (llama-3.3-70b → llama-3.3-70b-versatile) sem avisar o usuário. Tokens são consumidos no Groq mas o seletor mostra Cerebras. O usuário precisa ficar alternando manualmente.
IA sem treinamento personalizado — O system prompt está hardcoded em llm-system-prompt.ts. Não existe interface para admins adicionarem/editarem prompts de treinamento. Todas as LLMs usam o mesmo prompt fixo, sem possibilidade de customização pelo frontend.
Mensagens de erro genéricas sem identidade — Erros do assistente não mencionam o nome do sistema "Doc's Cataguases". Falta branding nas mensagens de erro e feedback ao usuário.

4. Plano de Implementação (Fases)

Visão Geral

6 módulos de melhoria divididos em 3 fases. Priorizados por impacto no usuário final. Cada fase é independente e entregável.

FASE 1 — Auto-Refresh + Notificações Inteligentes
Prioridade: CRÍTICA · Impacto: Todos os usuários
1
Sistema de Notificações por Usuário
Crítico API Web DB
  • Criar tabela Notificacao no Prisma com: id, userId (destinatário), tipo, mensagem, lida, portariaId, createdAt
  • Gerar notificações direcionadas:
    • Portaria rejeitada → notifica o autor
    • Portaria submetida → notifica revisores da secretaria
    • Revisão aprovada → notifica o autor
    • Aguardando assinatura → notifica PREFEITO/SECRETARIO
    • Publicada → notifica o autor
    • Usuário aprovado → notifica o próprio usuário
  • API: GET /api/notifications?userId=X com paginação
  • SSE: Filtrar por userId no polling, não apenas secretariaId
  • Frontend: Painel de notificações com mark-as-read, link direto p/ portaria
2
Auto-Refresh (eliminar F5)
Crítico Web
  • React Query refetchInterval em hooks chave:
    • use-dashboard.ts → polling 30s
    • use-portarias.ts (listagem) → polling 15s
    • Fila de revisão → polling 10s
    • Detalhe da portaria → polling 20s
  • SSE-driven invalidation: Quando SSE recebe evento relevante (ex: PORTARIA_SUBMETIDA), invalidar queries React Query via queryClient.invalidateQueries()
  • Tela /aguardando: Polling a cada 10s verificando se role mudou de PENDENTE
  • Window focus refetch: refetchOnWindowFocus: true no React Query global
3
Badges Dinâmicos na Sidebar
Alto API Web
  • API endpoint: GET /api/sidebar-counts retorna contagens filtradas por role:
    • REVISOR: portarias em EM_REVISAO_ABERTA da secretaria
    • PREFEITO/SECRETARIO: portarias em AGUARDANDO_ASSINATURA
    • ADMIN: usuários PENDENTE, portarias com FALHA
  • Hook: useSidebarCounts() com polling 15s + invalidação por SSE
  • Sidebar: Alimentar badge em NavLeaf items (Revisão, Assinatura, Usuários)
  • Animação: Pulse na badge quando valor muda (transição CSS)
FASE 1B — Correção e Evolução da IA/LLM
Prioridade: CRÍTICA · Impacto: Assistente IA + Experiência do Admin
4
Corrigir Seleção de Modelo / Failover
Crítico API Web
  • Bug principal: Quando Cerebras dá rate limit (429), o circuit breaker faz fallback silencioso para Groq, trocando modelo e provider sem avisar o usuário
  • Correção 1 — Respeitar escolha manual:
    • Se skipSmartRouter=true (usuário escolheu modelo), NÃO fazer failover automático
    • Retornar erro claro: "Doc's Cataguases: O modelo [X] via Cerebras está temporariamente indisponível. Tente outro modelo ou aguarde."
    • Manter failover automático apenas no modo "Automático"
  • Correção 2 — Transparência no failover:
    • No modo Automático, quando fizer fallback, retornar no response: { usedProvider, usedModel, fallbackReason }
    • Frontend exibe: "Respondido via Groq (Cerebras indisponível)" como nota discreta
  • Correção 3 — Erros com identidade:
    • Toda mensagem de erro do LLM prefixada com "Doc's Cataguases:"
    • Erros em pt-br consistentes: rate limit, timeout, sem chave, modelo indisponível
5
Página de Treinamento de IA (Prompts)
Crítico API Web DB
  • Tabela LlmPrompt no Prisma:
    • id, nome (identificador amigável)
    • categoria (SISTEMA, PORTARIA, REVISAO, CHAT_GERAL, MODELO, CUSTOM)
    • conteudo (texto do prompt)
    • ativo (boolean — pode desabilitar sem excluir)
    • ordem (prioridade de injeção)
    • criadoPor, criadoEm, atualizadoEm
  • Página admin /admin/ia/prompts:
    • Listar todos os prompts por categoria (accordion/tabs)
    • Criar novo prompt com editor de texto rico (textarea grande)
    • Editar/ativar/desativar prompts existentes
    • Preview: ver como ficaria o system prompt completo montado
    • Duplicar prompt para variações
    • Drag & drop para reordenar prioridade
  • Injeção dinâmica no chat:
    • buildSystemPrompt() monta prompt base + todos os prompts ativos do banco, ordenados por ordem
    • Prompts de categoria SISTEMA são sempre injetados
    • Prompts de PORTARIA injetados apenas no modo criação/revisão
    • Prompts CUSTOM injetados conforme contexto (secretaria, modo)
  • Aplicar a TODAS as LLMs: O system prompt montado é enviado independente do provider (Cerebras, Groq, Mistral, OpenRouter)

6. Status da Assinatura Atual

Veredicto: Trust-Based — Precisa de App Real de Assinatura

O sistema NÃO abre nenhum aplicativo de assinatura. Apenas marca como "assinado" no banco de dados baseado no login do usuário. Não mostra visualmente onde/como o documento foi assinado, e falta uma variável de "linha de assinatura" nos modelos de documento.

  • ✅ SHA-256 para integridade do PDF (real)
  • ✅ Registro de assinante, timestamp, tipo (real)
  • ✅ Upload de comprovante para assinatura manual (real)
  • ✅ Audit trail completo no feed de atividades (real)
  • ❌ Não abre app de assinatura (nenhuma integração externa)
  • ❌ Não mostra visualmente se foi assinado no documento
  • ❌ Falta variável {{LINHA_ASSINATURA}} nos modelos
  • ❌ Sem validação de certificado ICP-Brasil
  • ❌ Sem verificação de PDFs assinados externamente

Solução proposta (Fase 2, Módulo 9): Integrar com GOV.BR Assinatura (gratuito para governo), criar variável de linha de assinatura, e mostrar status visual claro no PDF e no frontend.
